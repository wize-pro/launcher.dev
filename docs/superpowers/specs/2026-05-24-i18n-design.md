# Internationalization (i18n) — Design

**Date:** 2026-05-24
**Status:** Approved
**Scope:** Make Dev Launcher an international, English-first project with switchable runtime languages.

## Goal

Two distinct outcomes:

1. **Repository is English-only (source).** Everything a visitor reads on GitHub is in English: `README.md`, `CLAUDE.md`, all code comments, console logs, docs (including this spec), example files. No French in the source.
2. **The running app is multilingual.** UI text and user-facing server messages are translatable at runtime. English is the base locale; French is the first additional locale. More languages can be added later by dropping in a JSON file — no code change.

These are complementary: the French strings currently hardcoded in the source become the seed for `locales/fr.json`, while the source itself switches to English.

## Non-goals

- No i18n library or framework (keeps the project's no-bundler / no-framework / vanilla-JS philosophy).
- No build step. Locale files are plain static JSON served as-is.
- No per-user accounts or server-side sessions (the tool is local, single-user).
- Pluralization/gender rules beyond simple `{var}` interpolation are out of scope for now.

## Architecture

### Translation files — single source of truth

```
locales/
├── en.json   # base / reference locale
└── fr.json   # first translation
```

- Flat, readable keys with dotted namespaces: `"settings.title"`, `"scan.found"`, `"error.projectNotFound"`.
- Values may contain `{name}`-style placeholders for interpolation.
- The **same files are consumed by both** the frontend (via `fetch('/locales/<lang>.json')`, served statically) and the backend (via `fs.readFileSync(path.join(__dirname,'locales',...))`). One source, no duplication.
- `en.json` is authoritative: every key that exists anywhere must exist in `en.json`. Missing keys in other locales fall back to the English value, then to the raw key.

### The `t()` helper (frontend and backend)

A tiny hand-rolled function (~15 lines), implemented once per side:

```
t(key, params)            // frontend: uses the currently active locale
t(key, lang, params)      // backend: caller passes the language
```

Behavior: look up `key` in the active/target locale → fall back to `en` → fall back to the key string itself. Then replace `{placeholders}` from `params`.

### Frontend wiring (`public/index.html`)

- Static HTML text carries a `data-i18n="key"` attribute. A single `applyTranslations()` pass fills every `[data-i18n]` element on load and on language change. Attributes like placeholders use `data-i18n-placeholder`, titles use `data-i18n-title`.
- Dynamically generated HTML (template literals) calls `t("key")` inline.
- A **language selector** sits in Settings next to the theme control.
- Changing language: update the in-memory locale, re-run `applyTranslations()`, re-render dynamic views — **no full page reload**.

### Backend wiring (`server.js`)

- Loads the locale JSON files once at startup into memory.
- The active language is read from `settings.json` (`lang` field).
- User-facing strings — API error messages (`res.json({ error: ... })`) and scan progress messages (the SSE `emit('found'|'warn'|...)` payloads) — go through `t(key, settings.lang, params)`.
- Console logs are **not** translated; they are written in English (part of the English-source requirement, not the runtime i18n).

## Data flow — choosing the language

- `settings.json` gains a `lang` field. Default: unset (absent).
- **First launch:** if `lang` is unset, the frontend detects `navigator.language` (`fr*` → `"fr"`, otherwise `"en"`), applies it, and persists it via `POST /api/settings`. A `localStorage` copy (`dlLang`) is written so the next load applies the right language instantly (avoids an English→French flash before settings load).
- **Subsequent loads:** read `dlLang` from `localStorage` for instant paint, then reconcile with `settings.json` (server is the source of truth).
- **Change:** user picks a language in Settings → `POST /api/settings` (persist) + update `localStorage` + re-render. The server now produces its messages in the new language.

## Components and responsibilities

| Unit | Responsibility | Depends on |
|---|---|---|
| `locales/*.json` | Hold all translatable strings | — |
| frontend `t()` + `applyTranslations()` | Render UI in the active language | active locale object |
| language selector (Settings) | Let the user switch language | `t()`, `/api/settings` |
| backend `t()` | Translate server-produced user-facing strings | `locales/*.json`, `settings.lang` |
| `settings.lang` (+ migration) | Persist the chosen language | settings schema |

## Schema impact

- `settings.json` gains `lang` (string | absent). Add it to `SETTINGS_DEFAULTS` as `null` (meaning "not yet chosen → detect").
- This is a backward-compatible additive field. It is covered by the existing schema-version mechanism; bump `CURRENT_SCHEMA_VERSION` only if a transform is needed (none is — absence is handled by first-launch detection), so no migration step is required.

## Error handling

- Missing locale file → fall back to `en`; if `en` is missing a key → return the key string. The app never crashes on a missing translation.
- Invalid `lang` value in settings → treated as unset → detection/`en` fallback.
- Locale JSON parse error at startup → log an English warning, continue with whatever loaded (at minimum `en`).

## Testing

- `t()` unit behavior: key found, key missing (fallback chain), placeholder interpolation.
- Locale completeness check: every key in `fr.json` exists in `en.json` and vice-versa (a small script; can run in CI later).
- Backend: an API error and a scan message return French text when `settings.lang === "fr"`, English otherwise.
- Manual: switch language in Settings, confirm UI updates without reload; restart app, confirm language persisted.

## Phased rollout

Each phase is independently shippable and testable.

- **Phase 0 — Infrastructure.** Create `locales/en.json` + `fr.json` (seeded with a handful of keys), the `t()` helpers (both sides), `applyTranslations()`, the `lang` settings field, first-launch detection, and the language selector. Nothing user-visible changes yet beyond the selector.
- **Phase 1 — UI strings.** Extract every French string from `public/index.html` into the locale files (English as base, French preserved). Wire `data-i18n` / `t()`. This is the largest, most mechanical phase.
- **Phase 2 — Server messages.** Route API error messages and scan SSE messages through `t()`.
- **Phase 3 — Repository English pass.** Translate to English: all code comments (`server.js`, `main.js`, `launcher.config.js`), console logs, `README.md`, `CLAUDE.md`, `example.launcher.yml`, `.gitignore` comments, and any remaining French in the repo. After this phase the repository reads as a fully English project.

## Open questions

None blocking. Future (post-1.0) considerations noted but out of scope: pluralization rules, RTL languages, translating the tray menu live (Electron menus are built once at startup — acceptable to follow `settings.lang` at build time for now).
